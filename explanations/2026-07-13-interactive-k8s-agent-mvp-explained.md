# Interactive Kubernetes Agent MVP 계획 쉬운 설명

원문: `docs/superpowers/plans/2026-07-13-interactive-k8s-agent-mvp.md`

## 한 줄 요약

이 계획은 **내 앱을 Kubernetes에 올릴 수 있게, 컴퓨터가 차근차근 물어보고 YAML 파일을 만들어 주는 도우미 프로그램**을 만들자는 설계도다.

## 무엇을 만들려는가

지금 프로젝트에는 이미 코드를 읽고 분석하는 `preanalyzer`가 있다.

예를 들면 이런 것을 알아낸다.

- 이 저장소는 어떤 앱인지
- 실행 명령은 무엇인지
- 포트는 몇 번인지
- 환경변수나 Secret이 필요한지
- Kubernetes로 만들 때 어떤 정보가 부족한지

이 문서는 그 위에 `k8sagent`라는 새 명령어 도우미를 만들자는 계획이다.

사용자는 대략 이런 흐름을 거친다.

```text
저장소 입력
→ 앱 분석
→ 어떤 컴포넌트를 배포할지 선택
→ 모르는 값은 질문받기
→ Kubernetes YAML 생성
→ 로컬에서 검증
→ 문제가 있으면 수정 제안
```

## 쉬운 비유

앱을 Kubernetes에 올리는 일은 이사 준비와 비슷하다.

`preanalyzer`는 집 안을 둘러보는 사람이다.

```text
침대 있음
책상 있음
냉장고 있음
전기 콘센트 필요함
```

`k8sagent`는 이사 도우미다.

```text
이 침대도 가져갈까요?
냉장고는 어느 방에 둘까요?
전기는 몇 개 필요할까요?
그럼 이삿짐 목록을 만들겠습니다.
목록이 맞는지 검사하겠습니다.
```

여기서 이삿짐 목록이 Kubernetes YAML이다.

## 중요한 원칙

LLM은 최종 Kubernetes YAML을 직접 만들지 않는다.

LLM은 설명, 질문 문장, 오류 설명, 수정 제안 정도만 도와준다. 실제 YAML은 Python 코드가 정해진 규칙대로 만든다.

이렇게 하는 이유는 **같은 입력이면 항상 같은 결과가 나와야 하기 때문**이다.

또 Secret 값, Git 토큰 같은 비밀은 로그, 파일, LLM 요청에 넣지 않는다. 비밀번호를 참조할 수는 있지만, 비밀번호 값을 만들어 내거나 저장하지 않는다.

## 어떤 순서로 만들 계획인가

계획은 17개 작업으로 나뉜다.

처음에는 `k8sagent` 패키지 뼈대를 만든다. 그다음 설정 파일, 세션 저장, Git 저장소 가져오기, 기존 분석기 연결을 만든다.

중간에는 사용자가 배포할 컴포넌트를 고르고, 질문에 답하고, 변경을 승인하는 구조를 만든다.

뒤쪽에는 Kubernetes YAML을 만드는 렌더러, 검증기, 실패 수정 제안, CLI 명령어, 대화형 마법사, 전체 테스트와 문서 업데이트가 있다.

즉 한 번에 대충 만드는 계획이 아니라, 작은 블록을 하나씩 테스트하면서 쌓는 계획이다.

## 핵심 결론

이 계획의 핵심은 다음과 같다.

**기존 분석기는 건드리지 않고, 그 위에 안전한 대화형 Kubernetes 배포 도우미를 새로 올린다.**
