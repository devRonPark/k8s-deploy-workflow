Interactive Kubernetes Agent MVP

이 변경은 기존의 결정론적 Kubernetes 사전분석기 위에, 사용자가 저장소를 넣고 질문에 답하면 안전하게 매니페스트까지 도달하도록 안내하는 새 대화형 에이전트의 첫 설계와 첫 테스트를 추가한다.

설명 대상: 현재 작업트리의 새 파일 핵심 파일: MVP 계획서, 쉬운 설명 문서, 설정 테스트 상태: 설계 중심 + 첫 실패 테스트

Background

Kubernetes 매니페스트 생성기는 보기보다 위험한 도구다. 저장소를 읽고 곧바로 YAML을 뱉는 방식은 편해 보이지만, 작은 추측 하나가 운영 설정이 된다. 포트를 잘못 고르면 서비스가 열리지 않고, Secret 값을 실수로 기록하면 보안 사고가 된다. 이 저장소의 기존 설계가 아주 강하게 반복하는 말은 그래서 하나다. 먼저 사실을 모으고, 그 사실에서만 결론을 내린다.

초심자용 큰 그림

현재 시스템의 이름은 `preanalyzer`다. 하는 일은 “애플리케이션 저장소를 읽어서 Kubernetes 배포에 필요한 후보 정보를 정리하는 것”이다. 중요한 점은 이 단계가 배포 버튼이 아니라 관찰과 정리의 단계라는 것이다.

기존 preanalyzer의 데이터 흐름

Repository Dockerfile, docker-compose.yml, package.json 같은 파일
Snapshot & Inventory 어떤 파일이 있는지, 어떤 commit을 봤는지 고정
Evidence “Dockerfile에 EXPOSE 8080이 있다” 같은 관찰 사실
Rules & Reconcile 규칙으로 후보를 만들고 충돌은 질문으로 남김
Intent & YAML 검증 가능한 중간 모델에서만 최종 YAML 생성

코드상으로 이 흐름은 `src/preanalyzer/pipeline.py`의 `run_phase1_analysis()`와 `run_analysis()`에 들어 있다. `run_phase1_analysis()`는 `00-repository-snapshot.yaml`부터 `04-semantic-analysis.yaml`까지 만들고, `run_analysis()`는 그 결과를 조정해서 Intent, 매니페스트, 검증 리포트까지 이어 간다. 

repo_snapshot = snapshot(...)
inventory = build_inventory(...)
evidence = build_evidence(...)
rules = infer(evidence)
semantic_audit, accepted_commands = _build_semantic_analysis_audit(...)

result = reconcile(rules, evidence, accepted_commands)
render = TemplateRenderer(...).render(intent)
report = ValidationPipeline().run(manifest_dir)

이번 변경과 직접 맞닿은 배경

이번 새 계획은 이 기존 흐름을 지우지 않는다. 대신 그 위에 `k8sagent`라는 별도 오케스트레이션 계층을 얹는다. 사용자는 저장소를 지정하고, 컴포넌트를 고르고, 빠진 값을 질문으로 채운다. 에이전트는 그 답변을 검증 가능한 모델 변경으로 바꾸고, Python 렌더러가 Kubernetes YAML을 만든다.

새 k8sagent가 놓이는 위치

기존 preanalyzer

  • 저장소를 결정론적으로 분석한다.
  • 근거, 후보, 질문, 기본 Intent를 만든다.
  • LLM이 최종 YAML을 만들지 못하게 한다.

새 k8sagent

  • 저장소 획득, 세션, 대화형 질문을 맡는다.
  • 사용자 답변을 검증된 ChangeSet으로 바꾼다.
  • 검증 실패 시 수정 제안을 승인 흐름으로 돌린다.
핵심 정의 “Intent”는 Kubernetes YAML 그 자체가 아니다. 어떤 컴포넌트를 어떤 이미지, 포트, 환경변수, 서비스로 배포할지 담는 중간 모델이다. 이 프로젝트의 중요한 규칙은 저장소에서 YAML로 바로 뛰지 않고, 항상 이런 중간 모델을 거친다는 점이다.

Intuition

이 변경의 직관은 간단하다. 기존 시스템은 눈이 좋다. 저장소를 보고 근거를 수집할 수 있다. 그러나 사용자의 의도를 묻고, 선택을 저장하고, 실패를 다시 수정으로 연결하는 진행자는 아직 없다. 새 에이전트는 그 진행자다.

장난감 예시: 작은 Node 앱

저장소에 다음 세 파일이 있다고 하자. 

Dockerfile:        EXPOSE 3000
package.json:      "start": "node server.js"
docker-compose.yml: ports: ["8080:3000"]

기존 분석기는 “컨테이너 안의 포트는 3000일 가능성이 높다”, “실행 명령은 node server.js다”, “호스트 8080은 외부 노출 포트일 수 있다” 같은 사실과 후보를 만든다. 그러나 배포 namespace, 이미지 registry, ingress host는 저장소 안에 없을 수 있다. 이때 정답을 꾸며내면 안 된다.

질문으로 남겨야 하는 값과 자동으로 채택할 수 있는 값

관찰됨 EXPOSE 3000, start command, component path
사용자에게 질문 registry, namespace, ingress host, 충돌하는 포트
검증된 Intent 근거 있는 값 + 사용자가 승인한 값만 포함

여기서 LLM의 역할은 답을 만드는 일이 아니다. “어떤 namespace를 쓰시겠습니까?”라는 문장을 더 자연스럽게 만들거나, kubeconform 오류를 사람이 읽기 쉽게 설명하거나, 자연어 요청을 ChangeSet 후보로 바꾸는 데 도움을 줄 수 있다. 하지만 최종 결정은 스키마와 승인 흐름을 통과해야 한다.

왜 별도 패키지인가

`preanalyzer`는 분석기다. 여기에 세션, Git clone 캐시, 대화형 REPL, 사용자 승인 루프를 넣으면 분석기와 제품 흐름이 섞인다. 이번 계획은 `k8sagent → preanalyzer` 방향의 단방향 의존만 허용한다. 말하자면 분석기는 도서관이고, 에이전트는 사서다. 책의 내용을 바꾸지 않고도 사용자가 필요한 장을 찾도록 안내할 수 있다.

Code

현재 작업트리에는 추적되지 않은 새 파일들이 있다. 그중 이 변경의 중심은 구현 코드가 아니라 구현 계획과 첫 단위 테스트다. 따라서 이 diff는 “완성된 에이전트”가 아니라 “어떤 에이전트를 어떤 안전 장치로 만들 것인가”를 고정하는 변경으로 읽어야 한다.

docs/superpowers/plans/2026-07-13-interactive-k8s-agent-mvp.md 새 `k8sagent`의 목표, 범위, 아키텍처 결정, 파일 구조, 17개 구현 태스크를 담은 중심 계획서.
tests/unit/agent/test_config.py Task 1의 첫 테스트. 아직 `src/k8sagent/` 구현이 없으므로, 다음 구현을 요구하는 실패 테스트 역할을 한다.
tests/unit/agent/__init__.py 새 agent 단위 테스트 패키지를 열기 위한 빈 초기화 파일.
explanations/2026-07-13-interactive-k8s-agent-mvp-explained.md MVP 계획을 비개발자도 읽을 수 있게 요약한 보조 설명 문서.

1. 계획서는 제품 경계부터 못 박는다

계획서의 목표는 “저장소 분석 → 컴포넌트 선택 → Kubernetes Intent 구축 → 질문/답변 → 매니페스트 생성 → 로컬 검증 → 수정 제안”까지 안내하는 CLI 에이전트다. 하지만 무엇을 하지 않을지도 분명히 한다. 새 의존성 추가, Secret 값 생성, Helm/Kustomize 지원, 실제 클러스터 배포, 무승인 자동 수정은 MVP 범위 밖이다.

결정 의미 왜 중요한가
새 패키지 `src/k8sagent/` 오케스트레이션을 분석기 밖으로 분리 `preanalyzer`의 결정론적 성격을 보존
LLM은 YAML 생성 금지 질문 문안, 설명, ChangeSet 후보까지만 허용 동일 입력에서 동일 출력이 나오는 재현성 보장
ChangeSet 승인 흐름 자연어 요청도 검증된 변경 후보와 diff를 거침 무심코 운영 설정이 바뀌는 일을 막음
산출물은 `k8s-agent-output/` 분석 대상 저장소 안에 결과를 쓰되 스캔 제외 예정 재분석 때 생성 YAML이 다시 evidence로 섞이지 않게 함

2. 첫 테스트는 설정 우선순위를 고정한다

`tests/unit/agent/test_config.py`는 아직 없는 `k8sagent.config.load_config()`가 어떤 동작을 해야 하는지 먼저 적는다. 이것은 테스트 주도 개발에서 흔히 쓰는 방식이다. 구현이 없으므로 지금은 import 단계에서 실패할 가능성이 높지만, 그 실패가 다음 작업의 시작점을 선명하게 만든다. 

cfg = load_config(env={}, home_override=Path("/tmp/x"))
self.assertEqual(cfg.k8s_version, "1.29")
self.assertTrue(cfg.llm_enabled)
self.assertEqual(cfg.git_token_env, "K8S_AGENT_GIT_TOKEN")
self.assertEqual(cfg.home, Path("/tmp/x"))

더 중요한 테스트는 우선순위다. 파일 설정보다 환경변수가 세고, 환경변수보다 CLI 값이 세다. 운영 도구에서는 이 순서가 자주 문제가 된다. 설정 출처가 명확하지 않으면 사용자는 “왜 내 값이 먹지 않았지?”라는 상태에 빠진다. 

(home / "config.yaml").write_text("k8s_version: '1.27'\n", encoding="utf-8")
env = {"K8S_AGENT_K8S_VERSION": "1.28"}
self.assertEqual(load_config(env=env, home_override=home).k8s_version, "1.28")

cfg = load_config(cli_overrides={"k8s_version": "1.30"}, env=env, home_override=home)
self.assertEqual(cfg.k8s_version, "1.30")

3. 오류 분류도 API의 일부가 된다

테스트는 `ConfigError("x").code == "config_error"`도 요구한다. 단순한 예외처럼 보이지만, CLI 제품에서는 오류 코드가 사용자 경험의 일부다. 나중에 `repo_acquisition_error`, `validation_run_error`, `llm_unavailable` 같은 오류를 exit code나 메시지로 매핑할 수 있기 때문이다. 

self.assertEqual(ConfigError("x").code, "config_error")
self.assertIsInstance(ConfigError("x"), AgentError)

4. 기존 코드와의 연결점

계획서가 기존 `preanalyzer`에서 빌려 쓰려는 중요한 자산은 세 가지다. 첫째, `run_phase1_analysis()`의 결정론적 분석 흐름. 둘째, `Tracked[T]`가 강제하는 값의 출처와 신뢰도 계약. 셋째, kubeconform과 kubectl dry-run으로 이어지는 검증 체인이다.

새 에이전트가 기존 자산을 소비하는 방식

k8sagent.repo local path 또는 Git URL을 분석 가능한 checkout으로 고정
preanalyzer snapshot, inventory, evidence, rules 생성
k8sagent.intent 사용자 선택과 질문 답변으로 확장된 Intent
k8sagent.render Python dict 렌더러가 YAML 생성
k8sagent.validate PASS, FAIL, PARTIAL로 결과 집계

Safety Notes

이 변경에서 가장 흥미로운 부분은 기능 목록이 아니라 경계 설정이다. Kubernetes 배포 도우미는 사용자가 편하게 느끼도록 해야 하지만, 편리함 때문에 검증이나 승인을 건너뛰면 안 된다.

Secret 값은 데이터 흐름에 들어오면 안 된다 계획서는 Git 토큰을 환경변수 이름으로만 다루고, Secret 값은 LLM 요청, 로그, 세션 파일, 산출물에 쓰지 않는다고 명시한다. 기존 `preanalyzer`도 Secret 후보 타입에 실제 값을 두지 않는 방식으로 같은 원칙을 지킨다.
현재 diff는 완성 기능이 아니다 `tests/unit/agent/test_config.py`는 새 패키지의 첫 계약을 적은 테스트다. 아직 `src/k8sagent/config.py`가 없으므로 이 테스트는 구현 전에는 실패하는 것이 자연스럽다.
검증 완료와 검증 생략은 다르다 기존 검증 파이프라인에서 `kubeconform: skipped`는 Kubernetes 스키마 검증이 끝났다는 뜻이 아니다. 새 에이전트 계획은 이를 `PASS`/`FAIL`/`PARTIAL` 집계로 더 분명하게 드러내려 한다.
