배경

이 브랜치 이전에도 분석기는 전반부를 꽤 신중하게 수행하고 있었습니다. 저장소를 스캔하고, Dockerfile·Compose 파일·패키지 매니페스트·Maven 파일 같은 산출물을 파싱한 뒤, 그 관측 결과를 증거와 결정론적 rule inference 후보로 바꿀 수 있었습니다. 다만 산출물은 04-semantic-analysis.yaml에서 멈췄습니다.

이 전반부가 중요한 이유는 프로젝트의 가장 중요한 불변식을 세우기 때문입니다. Kubernetes YAML은 저장소에서 바로 추측해서 만들면 안 됩니다. 분석기는 먼저 관측 사실을 기록하고, 그다음 중간 후보로 승격하며, 마지막에야 구조화된 intent 모델에서 매니페스트를 렌더링할 수 있습니다.

0스냅샷저장소 메타데이터, 커밋 해시, 작업트리 상태.
1인벤토리Dockerfile, Compose, 패키지 파일, Maven 파일.
2증거관측 사실. Secret 가능성이 있으면 안전하게 정리.
3규칙 후보컴포넌트, 런타임, 포트, 명령, 의존성 후보.
4Semantic audit활성화된 경우, 제한된 runtime-command agent 감사 기록.

이번 변경은 빠져 있던 후반부를 덧붙입니다. 새 산출물 범위는 05부터 15까지입니다. 여기에는 reconciliation report, component model, runtime model, dependency model, Kubernetes intent, unresolved questions, deployment profile echo, generated manifests, validation report, readiness checklist, smoke-test plan stub가 포함됩니다.

핵심 생각: 렌더러는 두 번째 분석기가 아닙니다. 일부러 단순하게 둡니다. 렌더러는 KubernetesIntent를 받고, 고정 템플릿을 적용하며, intent와 deployment profile이 근거를 제공하는 것만 렌더링합니다.

핵심 직관

단순한 매니페스트 생성기라면 “Dockerfile을 찾았다”에서 곧장 “Deployment를 만들자”로 뛰어갈 수 있습니다. 이 브랜치는 더 보수적인 길을 택합니다. 매니페스트 생성을 일련의 승격 단계로 보고, 각 승격 단계가 좁은 책임만 갖게 합니다.

간단한 입력

저장소에 EXPOSE 8000이 있는 Dockerfile, backend라는 Compose 서비스, 그리고 registry=reg.internal.demo, namespace=demo라고 적힌 profile이 있다고 해봅시다.

중간 intent

분석기는 backend라는 컴포넌트, 8000이라는 runtime port 후보, 그리고 workload image가 reg.internal.demo/backend:v0인 KubernetesIntent를 만듭니다.

렌더링 결과

렌더러는 backend/deployment.yaml, backend/service.yaml, backend/serviceaccount.yaml을 만들고, 필요하면 placeholder Secret도 만듭니다.

게이트들의 집합으로 보는 파이프라인

A관측결론 없이 사실을 기록합니다.
B추론규칙이 confidence와 evidence를 가진 후보를 만듭니다.
C조정충돌은 조용한 선택이 아니라 질문이 됩니다.
D렌더해결된 intent를 템플릿이 YAML로 바꿉니다.
E검증구문과 도구 결과가 달성 레벨을 결정합니다.

두 가지 작은 예가 이 설계를 잘 보여줍니다. entrypoint script가 실제 실행 명령을 숨기고 있다면 semantic agent가 그 스크립트를 살펴볼 수 있습니다. 하지만 매니페스트까지 흘러갈 수 있는 LLM 유래 값은 verifier가 수락한 command 하나뿐입니다. 또 Dockerfile은 8080을 말하고 Compose는 container port 8081을 말한다면, 시스템은 둘 중 하나를 고르지 않습니다. 대신 미해결 port 질문을 냅니다.

중요한 경계: placeholder Secret은 진짜 secret이 아닙니다. 매니페스트는 __REPLACE_ME__라는 리터럴 값을 사용합니다. 그래서 생성된 YAML은 리뷰에 쓸 수 있으면서도 secret 값 유출은 막습니다.

코드 흐름

1. 전체 오케스트레이터

pipeline.py에 새로 추가된 run_analysis 함수는 기존 Phase-1 준비 과정을 수행한 뒤 그 다음 단계까지 이어갑니다. rule 후보를 조정하고, 필요하면 deployment profile을 병합하고, 매니페스트를 렌더링하고, 검증한 다음 새 산출물을 씁니다.

semantic_audit, accepted_commands = _build_semantic_analysis_audit(...)
result = reconcile(rules, evidence, accepted_commands)

if profile_path is not None:
    profile = DeploymentProfile.model_validate(yaml.safe_load(...))
    merged = merge(result, profile)
    intent = merged.intent
    questions = merged.questions

render = TemplateRenderer(repo_snapshot.commit_sha, RULES_VERSION).render(intent)
report = ValidationPipeline().run(manifest_dir)

핵심은 순서입니다. semantic command는 reconciliation 전에 잡아두고, profile 값은 렌더링 전에 운영상 빈칸을 채우며, 검증은 실제 파일이 디스크에 생긴 뒤 실행합니다.

05–10Reconciliation report, component/runtime/dependency model, intent, unresolved questions.
12컴포넌트별로 정리된 generated manifest tree.
13–15Validation report, readiness checklist, smoke-test plan stub.

2. Semantic command 추출

audit 작성 로직은 이제 공개용 audit dictionary와 내부용 accepted command 목록을 함께 반환합니다. 이 차이는 작지만 중요합니다. audit에는 의도적으로 command text를 출력하지 않지만, reconciler는 검증된 command 값이 필요하기 때문입니다.

if str(result.verification_result.status) != "accepted":
    return None

candidate = next(c for c in result.resolution.candidates
                 if c.candidate_id == recommended_id)
return AcceptedSemanticCommand(
    component_id=candidate.component_id,
    command=str(candidate.value["command"]),
    evidence_refs=list(candidate.evidence_refs),
)

shell-entrypoint 데모 fixture는 scripted provider로 이 경로를 검증합니다. inspect_entrypoint_script를 호출해 semantic evidence를 받고, medium confidence의 llm_semantic_inference command를 내보냅니다. 그 값은 09-kubernetes-intent.yaml에 나타납니다.

3. 템플릿 렌더링

렌더러의 책임은 의도적으로 좁습니다. 컴포넌트를 정렬하고, dependency 컴포넌트는 건너뛰고, registry나 image name이 없으면 Deployment를 보류하며, 오직 템플릿에서만 리소스를 생성합니다.

if workload.image_registry is None or workload.image_registry.value is None:
    deferred.append(DeferredResource(component_id, "Deployment", "unresolved_image_registry"))
    continue

image = f"{registry}/{image_name}:{image_tag}"
files[f"{component_id}/deployment.yaml"] = render_template(...)

Labels와 annotations는 작은 policy 모듈에서 나옵니다. 렌더링된 모든 리소스에는 안정적인 app label과 commit SHA, rules version 같은 provenance annotation이 붙습니다. resource requests, probes, 그 밖의 운영 설정은 새로 지어내지 않습니다.

4. 검증

검증기는 세 단계를 실행하지만 외부 도구는 선택 사항으로 다룹니다. 깨진 YAML은 즉시 실패합니다. kubeconform이나 kubectl이 없으면 예외를 던지는 대신 skipped로 보고합니다. Achieved level 1이 되려면 YAML 구문과 kubeconform이 통과해야 합니다.

yaml_ok = self._yaml_syntax(manifest_dir, stages)
if yaml_ok:
    kubeconform_status = self._kubeconform(manifest_dir, stages)
else:
    stages.append(StageResult(stage="kubeconform", status="skipped"))

achieved = 1 if yaml_ok and kubeconform_status == "pass" else 0

5. CLI 진입점

새 CLI는 일부러 작게 만들었습니다. 기존 Phase-1 API를 바꾸지 않고 전체 파이프라인만 노출합니다. 사용자는 --no-llm으로 결정론적인 비-LLM 분석을 실행할 수 있고, 환경이 준비되어 있으면 OpenAI-compatible semantic mode를 요청할 수 있습니다.

python -m preanalyzer.cli analyze tests/fixtures/repos/node-express-like \
  --profile tests/fixtures/profiles/dev-profile.yaml \
  --no-llm --out /tmp/demo-out

6. Compose port를 runtime port 후보로 승격

port-conflict 데모가 한 가지 빈틈을 드러냈습니다. Compose port evidence는 있었지만, rule inference는 Dockerfile EXPOSE만 승격하고 있었습니다. 수정 후에는 해결된 Compose container port도 medium-confidence runtime port 후보로 승격합니다.

for fact in evidence.facts_by_type("compose_port"):
    service = fact.value.get("service")
    container_port = fact.value.get("container_port")
    if service in component_ids and isinstance(container_port, int):
        candidates.append(RuntimePortCandidate(service, container_port,
                                               fact.source, "medium", [fact.evidence_id]))

이 추가 후보 하나 덕분에 reconciliation은 8080과 8081을 함께 볼 수 있고, 그 차이를 질문으로 라우팅할 수 있습니다.

중요한 예외 상황

Shell entrypoint

Dockerfile에는 ENTRYPOINT ["./entrypoint.sh"]만 적혀 있을 수 있지만, Kubernetes에는 실제 runtime command가 필요합니다. 제한된 semantic task가 그 스크립트를 살펴볼 수 있고, verifier가 수락한 command만 다음 단계로 흐릅니다.

Port 충돌

Dockerfile과 Compose가 서로 다른 포트를 말할 수 있습니다. reconciler는 조용히 하나를 선호하면 안 됩니다. 생성된 runtime port는 null이 되고, 10-unresolved-questions.yaml에는 후보 목록이 들어갑니다.

Dockerfile 없음

JPetStore-like fixture는 Dockerfile이 없는 Maven 프로젝트가 억지로 Deployment로 렌더링되지 않는다는 점을 보여줍니다. rule에는 dockerfile_needed가 남고, 렌더링은 보류됩니다.

Secret 안전성

fastapi fullstack fixture에는 실제처럼 보이는 password 값이 들어 있습니다. acceptance 테스트는 전체 출력 트리를 스캔해 그 리터럴 값이 나타나지 않는지 확인합니다.
